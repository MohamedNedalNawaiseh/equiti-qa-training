Introduction to APIs

APIs (Application Programming Interfaces) are the backbone of modern software development. As a QA engineer, understanding APIs is essential for effective testing and automation.

Why APIs Matter for QA

  • APIs handle critical business logic - most bugs originate here
  • API testing is faster and more reliable than UI testing
  • You can test functionality before the UI is built
  • API defects are often security vulnerabilities

What is an API?

An API is a set of rules and protocols that allows different software applications to communicate with each other. Think of it as a waiter in a restaurant - you (the user) place an order, and the waiter (API) brings the food (data) from the kitchen (server).

Real-World API Example

User clicks "Get Weather" in an app
    ↓
App sends request to Weather API: GET /weather?city=Amman
    ↓
API server processes request
    ↓
API returns: { "temperature": 28, "condition": "Sunny", "humidity": 45 }
    ↓
App displays weather information

REST APIs

REST (Representational State Transfer) is the most common API architecture. REST APIs use standard HTTP methods to perform operations.

HTTP Methods

MethodPurposeExample
GETRetrieve dataGET /users - Get all users
POSTCreate new dataPOST /users - Create new user
PUTUpdate existing dataPUT /users/5 - Update user ID 5
PATCHPartial updatePATCH /users/5 - Update some fields
DELETEDelete dataDELETE /users/5 - Delete user 5

HTTP Status Codes

200 OK
201 Created
400 Bad Request
401 Unauthorized
403 Forbidden
404 Not Found
500 Server Error

Status Code Categories

  • 1xx: Informational (request received, continuing)
  • 2xx: Success (request successfully received and accepted)
  • 3xx: Redirection (further action needed)
  • 4xx: Client Error (bad syntax or cannot be fulfilled)
  • 5xx: Server Error (server failed to fulfill valid request)

Swagger UI

Swagger UI is an interactive documentation tool that lets you explore and test APIs directly from your browser. It reads the OpenAPI specification and generates a visual interface.

Swagger UI Features

  • Interactive Documentation: See all endpoints with descriptions
  • Try It Out: Execute API calls directly from the browser
  • Request Builder: Easily fill in parameters and body data
  • Response Viewer: View formatted responses with status codes
  • Authentication: Support for various auth methods

Endpoints & URL Structure

An endpoint is a specific URL where an API can be accessed. Each endpoint corresponds to a specific resource or operation.

URL Structure

https://api.example.com/v1/products/123?format=json&lang=en
\____/  \________/ \__/ \_______/ \__/ \_________/\________/
  |          |       |      |       |        |         |
 Scheme   Domain   Version Endpoint  ID    Query     Query
                                                   Parameter Value

Common Endpoint Patterns

# Collection endpoints
GET    /users          # Get all users
POST   /users          # Create new user

# Single resource endpoints
GET    /users/123      # Get user with ID 123
PUT    /users/123      # Full update of user 123
PATCH  /users/123      # Partial update of user 123
DELETE /users/123      # Delete user 123

# Nested resources
GET    /users/123/orders     # Get orders for user 123
GET    /users/123/orders/5   # Get order 5 for user 123

Request & Response Structure

Request Components

  • URL: The endpoint address
  • Method: GET, POST, PUT, PATCH, DELETE
  • Headers: Metadata (Content-Type, Authorization, etc.)
  • Query Parameters: Optional filters in URL (?page=1&limit=10)
  • Path Parameters: Resource identifiers in URL (/users/123)
  • Body: Data sent with request (for POST/PUT/PATCH)

Request Example

POST https://api.example.com/v1/users
Content-Type: application/json
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

{

}

Authentication

APIs use various authentication methods to secure endpoints and identify users.

Common Auth Methods

1. API Key

# API key passed in header
X-API-Key: your-api-key-here

# Or as query parameter
GET /users?api_key=your-api-key-here

2. Bearer Token (JWT)

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
# JWT has 3 parts: header.payload.signature

3. Basic Auth

Authorization: Basic base64(email:password)
# e.g., Basic dXNlckBleGFtcGxlLmNvbTpwYXNzd29yZA==

4. OAuth 2.0

# Authorization flow
1. Redirect user to: https://auth.example.com/authorize?client_id=...&redirect_uri=...
2. User logs in and grants permission
3. Server redirects back with: ?code=authorization_code
4. Exchange code for token:
   POST /token
   { "grant_type": "authorization_code", "code": "...", "client_secret": "..." }
5. Receive access token

Testing Auth APIs

Always test both positive (valid credentials) and negative (invalid/expired/missing tokens) authentication scenarios.

Parameters: Query, Path, and Body

Query Parameters

# Optional filters and pagination
GET /users?role=admin&status=active&page=2&limit=20

GET /products?category=electronics&price_min=100&price_max=500&sort=price_asc

Path Parameters

# Required resource identifiers
GET /users/123
GET /orders/456/items
GET /products/789/reviews

# Multiple path parameters
GET /companies/1/departments/5/employees

Request Body Parameters

POST /users
Content-Type: application/json

{
  "name": "Required field",
  "email": "Must be valid email",
  "age": 25,           // number
  "tags": ["qa","api"], // array
  "metadata": {         // nested object
    "source": "web",
    "referral": "google"
  }
}

Swagger Parameter Specification

parameters:
  - name: userId       # Parameter name
    in: path          # Location: path, query, header, cookie
    required: true    # Is it mandatory?
    schema:
      type: integer
    description: "The unique user identifier"
    
  - name: page
    in: query
    required: false
    schema:
      type: integer
      default: 1
    description: "Page number for pagination"

API Testing Strategies

Test Categories

  • Functional Testing: Does the API work as expected?
  • Schema Validation: Does response match the expected structure?
  • Boundary Testing: Edge cases like empty strings, max lengths
  • Security Testing: Auth, authorization, data exposure
  • Performance Testing: Response time under load
  • Error Handling: Invalid inputs, server errors

Example Test Cases

TEST CASE: TC-API-001 - Get User by Valid ID
------------------------------------------
Request:  GET /users/123
Expected: 
  - Status: 200 OK
  - Body contains: { "id": 123, "name": "...", "email": "..." }
  - Response time < 500ms

TEST CASE: TC-API-002 - Get User by Invalid ID
----------------------------------------------
Request:  GET /users/999999
Expected:
  - Status: 404 Not Found
  - Body contains: { "error": "User not found", "code": 404 }

TEST CASE: TC-API-003 - Create User with Missing Required Fields
----------------------------------------------------------------
Request:  POST /users
          Body: { "name": "Test" }  // missing email
Expected:
  - Status: 400 Bad Request
  - Body contains: { "error": "Email is required" }

TEST CASE: TC-API-004 - Create User with Invalid Email
------------------------------------------------------
Request:  POST /users
          Body: { "name": "Test", "email": "not-an-email" }
Expected:
  - Status: 400 Bad Request
  - Body contains: { "error": "Invalid email format" }

API Testing Best Practices

Test Independently

Each test should setup its own data and clean up after

Validate Schema

Always check response structure matches the contract

Test Edge Cases

Empty arrays, null values, max lengths, special characters

Chain Requests

Use response data from one call as input for next

Hands-On Exercises

Exercise 1: Explore a Public API

Task: Using Swagger UI or Postman, explore the JSONPlaceholder API:

  • GET https://jsonplaceholder.typicode.com/users - Get all users
  • GET /users/1 - Get user with ID 1
  • GET /users?username=Bret - Filter by username
  • POST /posts - Create a new post

Deliverable: Document responses for each endpoint with status codes and sample data.

Exercise 2: Test Authentication Flow

Task: Test a mock login API with various scenarios:

  • Valid credentials → expect 200 + token
  • Invalid password → expect 401 + error message
  • Non-existent user → expect 401 or 404
  • Empty credentials → expect 400 validation error
  • Missing Content-Type header → expect 415

Exercise 3: Create Comprehensive Test Suite

Task: Build a test suite for a CRUD API with:

  • Create (POST) - Test with valid data, missing fields, invalid data types
  • Read (GET) - Test existing ID, non-existent ID, multiple records
  • Update (PUT/PATCH) - Full update, partial update, invalid fields
  • Delete (DELETE) - Existing ID, non-existent ID
