What is Playwright?

Playwright is a powerful Node.js library for browser automation developed by Microsoft. It allows you to control Chromium, Firefox, and WebKit browsers with a single API.

Why Playwright?

  • Cross-browser: Test on Chromium, Firefox, and WebKit with one API
  • Fast & Reliable: Auto-waiting, network interception, parallel testing
  • TypeScript Support: Built-in TypeScript support, excellent IDE autocomplete
  • Powerful Features: Screenshots, videos, tracing, mocking
  • Modern Architecture: Designed for modern web apps (SPA, PWAs)
  • Active Development: Backed by Microsoft, frequent updates

Key Concepts

  • Browser: Instance of Chromium, Firefox, or WebKit
  • BrowserContext: Isolated session within a browser (like incognito)
  • Page: Single tab or popup window
  • Frame: iframe within a page
  • Locator: Way to find elements on a page

Browser, Context, Page Relationship

Browser (Chromium)
├── BrowserContext 1 (New User Session)
│   ├── Page 1 (Main Tab)
│   ├── Page 2 (Popup)
│   └── Frame (iframe in Page 1)
├── BrowserContext 2 (Another User Session)
│   └── Page 3
└── BrowserContext 3 (Admin Session)
    └── Page 4

Locators & Selectors

Locators are how Playwright finds elements on a page. Choosing the right locator is crucial for reliable tests.

Locator Types (Best to Worst)

// 1. ROLE LOCATOR (Recommended - accessible, semantic)
await page.getByRole('button', { name: 'Submit' }).click()
await page.getByRole('textbox', { name: 'Email' }).fill('test@example.com')
await page.getByRole('checkbox', { checked: true })

// 2. TEXT LOCATOR
await page.getByText('Welcome back').click()
await page.getByLabel('Password').fill('secret')

// 3. TEST ID LOCATOR (if data-testid attribute is set)
await page.getByTestId('submit-button').click()

// 4. CSS SELECTOR (when above don't work)
await page.locator('.btn-primary').first().click()
await page.locator('#login-form input[name="email"]').fill('test@example.com')

// 5. XPATH (last resort - brittle)
await page.locator('xpath=//button[contains(text(), "Submit")]').click()

Creating Locators

const locator = page.locator('selector');

// Chain locators for specificity
const form = page.locator('#login-form');
const emailInput = form.locator('input[name="email"]');

// Find elements within a locator
const items = page.locator('.item');
await items.first().click();      // First item
await items.last().click();       // Last item
await items.nth(2).click();       // Third item
await items.count();              // Get total count

Best Practices for Locators

✅ DO

  • Use semantic locators (role, label, text)
  • Add data-testid for critical elements
  • Use chained locators for context
  • Prefer locators over XPath

❌ DON'T

  • Use fragile CSS like :nth-child
  • Use indices (they change!)
  • Use long XPath expressions
  • Select by random generated classes

User Interactions

Playwright provides intuitive methods for simulating user actions.

Common Actions

// Clicking
await page.getByRole('button', { name: 'Submit' }).click();
await page.locator('#btn').dblclick();
await page.locator('#btn').click({ button: 'right' }); // Right click

// Typing
await page.getByLabel('Email').fill('test@example.com');
await page.getByRole('textbox').pressSequentially('hello', { delay: 100 });

// Clearing
await page.getByLabel('Search').clear();

// Checkboxes & Radio buttons
await page.getByRole('checkbox', { name: 'Remember me' }).check();
await page.getByRole('radio', { name: 'Premium Plan' }).selectRadio();

// Dropdowns (select element)
await page.locator('select#country').selectOption('jordan');
await page.locator('select#country').selectOption({ label: 'Jordan' });
await page.locator('select#country').selectOption(['jo', 'ae']); // Multiple

// Hover & Focus
await page.locator('.dropdown').hover();
await page.getByLabel('Email').focus();

Keyboard & Mouse

// Keyboard
await page.keyboard.press('Enter');
await page.keyboard.press('Control+A');
await page.keyboard.type('Hello World');
await page.keyboard.down('Shift');
await page.keyboard.up('Shift');

// Mouse
await page.mouse.click(100, 200);
await page.mouse.dblclick(100, 200);
await page.mouse.move(100, 200);
await page.mouse.down();
await page.mouse.up();

// Drag and Drop
await page.locator('#source').dragTo(page.locator('#target'));

File Upload/Download

// Upload file
await page.locator('input[type="file"]').setInputFiles('path/to/file.pdf');
await page.locator('input[type="file"]').setInputFiles([
  'file1.pdf',
  'file2.pdf',
]);
// Clear file selection
await page.locator('input[type="file"]').setInputFiles([]);

// Download (setup before action)
const [download] = await Promise.all([
  page.waitForEvent('download'),
  page.locator('#download-btn').click()
]);
await download.saveAs('path/to/save/' + download.suggestedFilename());

Assertions & Waits

Playwright has smart auto-waiting - it waits for elements to be actionable before performing actions. But you still need assertions to verify expected behavior.

Built-in Assertions

// Expect library (recommended)
import { expect } from '@playwright/test';

// Basic assertions
await expect(page.locator('.title')).toHaveText('Welcome');
await expect(page.locator('.title')).not.toHaveText('Error');
await expect(page.locator('#count')).toContainText('5 items');

// Visibility
await expect(page.getByRole('button')).toBeVisible();
await expect(page.getByRole('button')).toBeHidden();
await expect(page.getByRole('button')).toBeEnabled();
await expect(page.getByRole('button')).toBeDisabled();

// Form fields
await expect(page.getByLabel('Email')).toHaveValue('test@example.com');
await expect(page.getByLabel('Accept Terms')).toBeChecked();

// Counts
await expect(page.locator('.item')).toHaveCount(5);
await expect(page.locator('.item')).toHaveLength(5);

// Custom messages
await expect(page.locator('.error')).toHaveText('Invalid input', {
  timeout: 5000
});

Waiting for Conditions

// Auto-wait (built into every action)
await page.click('#btn'); // Waits for button to be visible & enabled

// Explicit waits (when needed)
await page.waitForSelector('#loaded-content', { state: 'visible' });
await page.waitForSelector('#loading', { state: 'hidden' });
await page.waitForURL('**/dashboard');
await page.waitForResponse(response => response.url().includes('/api/data'));

// Wait for navigation
await page.click('#btn');
await page.waitForURL('**/success');

// Wait for function
await page.waitForFunction(() => window.innerWidth > 768);

// Network idle (wait for no network for 500ms)
await page.waitForLoadState('networkidle');

Common Wait Patterns

// ❌ BAD - Never use arbitrary sleeps
await page.waitForTimeout(5000); // This is a code smell!

// ✅ GOOD - Wait for specific conditions
// Wait for element to appear
await expect(page.getByText('Success')).toBeVisible();

// Wait for API response
const response = await page.waitForResponse(
  r => r.url().includes('/api/users') && r.status() === 200
);

// Wait for function to return true
await page.waitForFunction(() => document.querySelectorAll('.item').length > 0);

Page Object Model

Page Object Model (POM) is a design pattern that creates an object-oriented representation of pages, making tests more maintainable and reducing code duplication.

Page Object Example

// pages/LoginPage.ts
import { Page, Locator, expect } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly submitButton: Locator;
  readonly errorMessage: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput = page.getByLabel('Email');
    this.passwordInput = page.getByLabel('Password');
    this.submitButton = page.getByRole('button', { name: 'Sign In' });
    this.errorMessage = page.locator('.error-message');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
  }

  async expectError(message: string) {
    await expect(this.errorMessage).toHaveText(message);
  }
}

Using Page Objects in Tests

// tests/login.spec.ts
import { test, expect } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';
import { DashboardPage } from '../pages/DashboardPage';

test('user can login with valid credentials', async ({ page }) => {
  const loginPage = new LoginPage(page);
  const dashboardPage = new DashboardPage(page);

  await loginPage.goto();
  await loginPage.login('test@example.com', 'password123');

  await expect(dashboardPage.welcomeMessage).toBeVisible();
  await expect(dashboardPage.welcomeMessage).toContainText('Welcome');
});

test('shows error with invalid credentials', async ({ page }) => {
  const loginPage = new LoginPage(page);

  await loginPage.goto();
  await loginPage.login('invalid@example.com', 'wrongpass');

  await loginPage.expectError('Invalid email or password');
});

API Testing with Playwright

Playwright can also make direct HTTP requests without a browser, perfect for API testing.

Making API Requests

import { test, request } from '@playwright/test';

test('API - Get user list', async ({ }) => {
  const apiContext = await request.newContext({
    baseURL: 'https://api.example.com',
    extraHTTPHeaders: {
      'Authorization': `Bearer ${process.env.API_TOKEN}`,
    },
  });

  const response = await apiContext.get('/users');
  expect(response.status()).toBe(200);
  
  const users = await response.json();
  expect(users).toHaveLength(10);
  expect(users[0]).toHaveProperty('id');
  expect(users[0]).toHaveProperty('email');
});

test('API - Create new user', async ({ }) => {
  const apiContext = await request.newContext({
    baseURL: 'https://api.example.com',
  });

  const newUser = {
    name: 'Test User',
    email: `test${Date.now()}@example.com`,
    role: 'user',
  };

  const response = await apiContext.post('/users', {
    data: newUser,
    headers: {
      'Content-Type': 'application/json',
    },
  });

  expect(response.status()).toBe(201);
  
  const createdUser = await response.json();
  expect(createdUser).toMatchObject(newUser);
  expect(createdUser).toHaveProperty('id');
});

Combining API and UI Testing

test('UI + API - Create user and verify in UI', async ({ page, request }) => {
  // 1. Create user via API
  const apiContext = await request.newContext({
    baseURL: 'https://api.example.com',
  });

  const newUser = {
    name: 'John Doe',
    email: 'john@example.com',
  };

  const createResponse = await apiContext.post('/users', { data: newUser });
  const created = await createResponse.json();

  // 2. Login as admin in UI
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('admin@example.com', 'adminpass');

  // 3. Navigate to users list
  const usersPage = new UsersPage(page);
  await usersPage.goto();

  // 4. Verify user appears in UI
  await expect(usersPage.getUserRow(created.id)).toBeVisible();
  await expect(usersPage.getUserEmail(created.id)).toHaveText(newUser.email);
});

Playwright Best Practices

✅ Use Page Objects

Encapsulate page logic in classes, don't use selectors directly in tests

✅ Meaningful Test Names

test('should show error when password is empty') not test('test1')

✅ Use Data-TestIDs

Add data-testid attributes for critical elements that change often

✅ Clean Up Test Data

Delete or reset data created during tests to avoid pollution

✅ Parallel Execution

Run tests in parallel to reduce execution time

✅ Retry Flaky Tests

Use retries for infrastructure issues, fix test if it fails consistently

Common Pitfalls

// ❌ BAD PRACTICES
// 1. Using sleep instead of proper waits
await page.waitForTimeout(3000);

// 2. Using generic selectors
await page.locator('div > div:nth-child(3) > span').click();

// 3. Not cleaning up state between tests
// 4. Hardcoding data instead of using fixtures
// 5. Not using TypeScript types

// ✅ GOOD PRACTICES
// 1. Use built-in auto-waiting
await page.getByRole('button').click();

// 2. Use semantic selectors
await page.getByRole('button', { name: 'Submit' }).click();

// 3. Use test.beforeEach for setup
// 4. Use fixtures for test data
// 5. Use TypeScript for type safety
