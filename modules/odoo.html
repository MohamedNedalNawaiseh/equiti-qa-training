Introduction to ERP Systems

Enterprise Resource Planning (ERP) systems integrate core business processes into a unified system. Odoo is one of the most popular open-source ERP platforms, used by businesses worldwide for managing operations.

Why ERP Testing is Important

  • Business Critical: ERP handles core business data and processes
  • Data Integrity: Errors can corrupt financial, inventory, or customer data
  • Integration: ERP connects multiple departments (sales, inventory, HR, accounting)
  • Compliance: Must meet regulatory requirements (tax, accounting standards)
  • Customization: Each implementation is customized, requiring thorough testing

Key ERP Modules (Odoo)

  • CRM: Customer relationship management, lead/opportunity tracking
  • Sales: Quotation, sales orders, invoicing
  • Purchase: Vendor management, purchase orders
  • Inventory: Warehouse management, stock tracking
  • Accounting: Financial transactions, reporting
  • HR: Employee management, payroll
  • Manufacturing: Production planning, BOMs

Business Workflows in ERP

Vendor → Purchase Order → Goods Receipt → Quality Check → Inventory

SALES WORKFLOW:
Lead → Opportunity → Quotation → Sales Order → Delivery → Invoice → Payment

INVENTORY WORKFLOW:
Receive → Store → Pick → Pack → Ship → Track

ACCOUNTING WORKFLOW:
Transaction → Journal Entry → Ledger → Financial Report → Audit

Users & Access Control

User Types in Odoo

User TypeAccess
AdministratorFull system access, settings, technical menu
EmployeeBasic access based on app permissions
Portal UserLimited external access (customers, vendors)
Public UserWebsite-only access

Access Control Levels

ACCESS CONTROL HIERARCHY:
1. Global Permissions (user type based)
   ├── Can create/read/write/delete
   └── App-specific restrictions

2. Record Rules (row-level security)
   ├── Users see only their records
   ├── Departments see their records
   └── Managers see team records

3. Field Level Security
   ├── Hide fields from certain users
   └── Make fields read-only

4. Menu Visibility
   ├── Apps shown based on installed modules
   └── Custom menu restrictions

Testing Access Control

TEST CASES FOR ACCESS CONTROL:
───────────────────────────────────────────────────────
1. Admin can access all features
2. Employee can only access permitted apps
3. User cannot access records outside their scope
4. Portal user can only see their own quotations/orders
5. Hidden fields are not visible to restricted users
6. Deleted records are not accessible
7. Menu items are shown/hidden correctly

Testing the Sales-to-Invoice Flow

END-TO-END TEST: SALES ORDER TO INVOICE
═══════════════════════════════════════════════════════
1. Create Quotation
   - Add customer, products, quantities
   - Verify prices and taxes calculated
   - Confirm quotation has expected total

2. Confirm Quotation → Sales Order
   - Click "Confirm Order"
   - State changes from Quotation to Sales Order
   - Reserved stock in inventory

3. Validate Delivery
   - Confirm delivery order
   - Update quantities delivered
   - Verify stock moves recorded

4. Create Invoice
   - Click "Create Invoice"
   - Select "Regular" or "By Delivery" invoicing
   - Verify invoice matches delivery

5. Validate Invoice
   - Post the invoice
   - Verify accounting entries created
   - Check customer balance updated

Common Test Scenarios

Scenario 1: Purchase Order Workflow

TEST CASE: TC-PO-001 - Complete Purchase Order Process
───────────────────────────────────────────────────────
Prerequisites:
- Vendor "ABC Supplies" exists with contact info
- Product "Laptop Pro" exists with price $1200
- Current stock: 0 units

Steps:
1. Create Purchase Order for vendor ABC Supplies
2. Add product Laptop Pro, quantity: 5
3. Confirm Purchase Order
4. Validate incoming shipment
5. Receive 5 units into stock
6. Verify vendor bill can be created
7. Create and post vendor bill
8. Verify accounting entries are correct

Expected Results:
- PO status: Purchase Order
- Inventory: 5 units of Laptop Pro
- Stock value increased by $6000
- Vendor bill posted: $6000 credit to Accounts Payable

Scenario 2: Inventory Valuation

TEST CASE: TC-INV-001 - Inventory FIFO Valuation
───────────────────────────────────────────────────────
Prerequisites:
- Product "Widget A" uses FIFO costing
- Starting stock: 10 units @ $10 = $100

Steps:
1. Receive 20 units @ $12 = $240 (2 Purchase Orders)
2. Deliver 15 units to customer
3. Check inventory valuation

Expected Results:
- Remaining stock: 15 units
- Cost calculation (FIFO):
  - 10 units @ $10 = $100
  - 5 units @ $12 = $60
- Total inventory value: $160

Scenario 3: Access Control Test

TEST CASE: TC-SEC-001 - Salesperson Access Restriction
───────────────────────────────────────────────────────
Prerequisites:
- User "Sales Rep" with role: Sales / Salesperson
- User assigned to "West Region" sales team
- Customer "Customer West" assigned to West Region

Steps:
1. Login as Sales Rep
2. Try to access customer from East Region
3. Try to access quotation from East Region
4. Access customer "Customer West"

Expected Results:
- Cannot see East Region customers/quotes
- Can see and modify own team's records
- Cannot access Administrator settings
