Introduction to Software Testing

Software testing is the process of evaluating a software application to identify defects, verify functionality, and ensure it meets requirements. As a QA engineer, your role is to ensure quality before release.

Why Testing Matters

  • Quality Assurance: Delivers bug-free software to customers
  • Cost Savings: Finding bugs early is 10x cheaper than in production
  • Customer Satisfaction: Reliable software builds trust
  • Security: Prevents vulnerabilities and data breaches
  • Compliance: Meets industry standards and regulations

Testing Types Overview

TypeDescriptionWhen
Smoke TestingQuick checks if critical features workAfter every build
Sanity TestingFocused tests on specific featuresAfter bug fixes
Regression TestingEnsure existing features still workBefore releases
Functional TestingTest each feature against requirementsDuring test cycles
Integration TestingTest combined modules togetherAfter unit tests
UATUser acceptance testingBefore production

Software Development Life Cycle (SDLC)

SDLC is the process of creating software from conception to deployment. Understanding SDLC helps you know when and how to test.

SDLC Phases

┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  Planning   │───▶│    Design   │───▶│ Development │───▶│   Testing   │
│  & Analysis │    │             │    │             │    │             │
└─────────────┘    └─────────────┘    └─────────────┘    └──────┬──────┘
       │                                                           │
       │    ┌─────────────┐    ┌─────────────┐    ┌──────────────┘
       └───▶│  Deployment │◀───│  Fix Bugs   │◀───┘
            │  & Maint.   │    │             │
            └─────────────┘    └─────────────┘

Phase Details

  • 1. Planning & Analysis: Requirements gathering, feasibility study, project plan
  • 2. Design: System architecture, UI design, database design, test plan creation
  • 3. Development: Coding, code reviews, unit testing
  • 4. Testing: Test execution, bug reporting, regression testing
  • 5. Deployment: Release to production, user training
  • 6. Maintenance: Bug fixes, updates, performance monitoring

Testing in SDLC

SDLC PhaseTesting Activity

DesignReview test plans, identify testable items
DevelopmentVerify unit tests, prepare test data
TestingExecute test cases, report bugs
DeploymentFinal verification, smoke tests in production

Software Testing Life Cycle (STLC)

STLC is the testing-specific process that runs parallel to SDLC. It has defined entry and exit criteria for each phase.

STLC Phase Details

  • Requirement Analysis: Analyze requirements, identify testable features, prepare RTM
  • Test Planning: Create test plan, select testing types, estimate effort
  • Test Case Development: Write test cases, peer review, create test data
  • Environment Setup: Setup test environment, configure tools, prepare smoke tests
  • Test Execution: Execute tests, log defects, retest fixed bugs
  • Test Closure: Analyze results, prepare test summary report

Entry vs Exit Criteria

Entry Criteria: Conditions that must be met before testing begins (e.g., test plan approved, test environment ready)

Exit Criteria: Conditions that must be met to conclude testing (e.g., all test cases executed, critical bugs fixed)

Writing Test Cases

A test case is a document that describes a specific test scenario, including inputs, actions, and expected results. Well-written test cases are crucial for effective testing.

Test Case Template

FieldDescription
Test Case IDUnique identifier (e.g., TC-LOGIN-001)
Test DescriptionWhat the test verifies
PrerequisitesPre-conditions that must be met
Test StepsStep-by-step instructions
Test DataData to use in the test
Expected ResultWhat should happen
Actual ResultWhat actually happened (filled during execution)
StatusPass/Fail/Blocked
PriorityP0, P1, P2, P3 (P0 = highest)

Good vs Bad Test Case Examples

✅ GOOD Test Case

ID: TC-LOGIN-001
Title: Verify user can login with valid credentials

Prerequisites:
- User must be registered with valid email and password
- Application must be accessible

Steps:
1. Navigate to login page
2. Enter "testuser@example.com" in email field
3. Enter "Test@123" in password field
4. Click "Sign In" button

Expected Result:
User should be logged in successfully and
redirected to dashboard within 3 seconds.
Welcome message should display "Welcome, Test User"

❌ BAD Test Case

ID: TC-LOGIN-002
Title: Login test

Steps:
1. Go to login page
2. Enter some credentials
3. Click login

Expected Result:
It should work

Test Case Design Techniques

  • Equivalence Partitioning: Divide inputs into groups that behave similarly
  • Boundary Value Analysis: Test edge cases (min, max, just inside, just outside)
  • Decision Table Testing: Test combinations of conditions
  • State Transition Testing: Test how the system changes states
  • Error Guessing: Use experience to guess where bugs might occur

Testing Techniques

1. Black Box Testing

Testing based on requirements without knowing internal code structure. Focus on inputs and outputs.

2. White Box Testing

Testing with knowledge of internal code structure. Used by developers for unit testing.

3. Gray Box Testing

Combination of black and white box. Tester has limited knowledge of internals.

Exploratory Testing

Simultaneous learning, test design, and execution. Particularly useful when:

  • Documentation is incomplete
  • Time is limited
  • Exploring a new feature
  • Finding unexpected bugs
CHARTER FOR EXPLORATORY SESSION:
"Explore the checkout flow to verify discount codes
are applied correctly to the final total"

TIMEBOX: 30 minutes
COVERAGE AREAS:
- Apply valid discount code
- Apply invalid discount code
- Expired discount code
- Multiple discount codes
- Discount code case sensitivity

Agile & Scrum for QA

Most modern software development uses Agile methodologies. Understanding Agile is essential for QA engineers.

Agile Manifesto Principles

  • Individuals and interactions over processes and tools
  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan

Scrum Roles

RoleResponsibility
Product OwnerOwns the backlog, prioritizes features, defines requirements
Scrum MasterFacilitates Scrum events, removes impediments
Development TeamCross-functional team including QA, devs, designers

Scrum Events

  • Sprint Planning: Plan work for the sprint (2 weeks)
  • Daily Standup: 15-minute sync, what did I do, what will I do, blockers?
  • Sprint Review: Demo completed work to stakeholders
  • Sprint Retrospective: Team reflection on how to improve

Jira Workflow for QA

Jira is the most popular issue tracking tool. Understanding Jira workflows is essential for bug reporting and tracking.

Typical Jira Workflow

To Do → In Progress → In Review → Testing → Done
  │          │              │          │
  │          │              │          └── QA verifies fix
  │          │              │
  │          │              └── Dev reviews code
  │          │
  │          └── Developer working on fix
  │
  └── Newly created issue

Issue Types in Jira

  • Epic: Large feature spanning multiple sprints
  • Story: User-facing feature with business value
  • Task: Technical work not user-facing
  • Bug: Defect in existing functionality
  • Subtask: Breakdown of larger issues

Creating a Good Bug Ticket

Summary: [What] happens when [Trigger]
             "Login fails when password contains @ symbol"

Steps to Reproduce:
1. Navigate to login page
2. Enter valid username
3. Enter password with @ symbol (e.g., Pass@word123)
4. Click Sign In

Expected: User should login successfully
Actual: Error "Invalid credentials" displayed

Environment: Chrome 120, Windows 11
Severity: Major (affects users with @ in password)
Priority: High
Attachments: screenshot, logs, video
